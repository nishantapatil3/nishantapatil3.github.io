<p><a href="https://spiffe.io/">spiffe.io</a> is a universal identity control plane for distributed systems.</p><p>SPIFFE is a set of standards to help us achieve zero trust identity framework. And SPIRE is one of the implementation of SPIFFE. SPIRE can authenticate and authorize workloads in a distributed system.</p><p>SPIRE can be deployed in variours methods. Ex: Single trust domain, Federation and Nested. SPIRE Federation is one of the deployment architectures of SPIFFE/SPIRE. <a href="https://spiffe.io/docs/latest/architecture/">https://spiffe.io/docs/latest/architecture/</a></p><figure><img alt="" src="images/1-bgrefmggs2uza7aasmtdfg.png" /></figure><p>To set up spire federation for two different trust domains, we need to set up spire-server and agent in two different platforms and federate them with each other using external IP address obtained by load balancers.</p><p>In our case let us setup spire federation in Kubernetes clusters by deploying spire components and two workloads server and client in both of the kind clusters.</p><p>Generally, we establish connection using grpc.dial which is insecure and end up with non-encrypted communication between workloads, this can be secured using mTLS connection made along with grpc.dial connection.</p><h3>Spire Federation Example using Spiffe Authentication</h3><p>Using an example from SPIRE official user guide for Federation model added here.</p><p><a href="https://github.com/spiffe/spire-tutorials/tree/master/docker-compose/federation">spire-tutorials/docker-compose/federation at master · spiffe/spire-tutorials</a></p><p>I modified the server-client example to be deployable in Kubernetes clusters with SPIRE server-agent model. This example demonstrates a simple a stock quote webapp frontend and service backend by setting up two SPIFFE-identified workloads that are identified by two different SPIRE Servers.</p><p>Let’s say we have a stock broker’s webapp that wants to display stock quotes fetched from a stock market web service provider. The scenario goes as follows:</p><ol><li>The user enters the broker’s webapp stock quotes URL in a browser.</li><li>The webapp workload receives the request and makes an HTTP request for quotes to the stock market service using mTLS.</li><li>The stock market service receives the request and sends the quotes in the response.</li><li>The webapp renders the stock quotes page using the returned quotes and sends it to the browser.</li><li>The browser displays the quotes to the user. The webapp includes some JavaScript to refresh the page every 1 second, so every second these steps are executed again.</li></ol><p>I have pushed the codes to below repository. Server hold the set of secure information and client exposes an http endpoint that we can open up in a web browser. The connection between Server-Client is gRPC protocol and we’ll secure this connection by setting up mTLS connection using SPIFFE identity framework.</p><p>First, let’s clone below repository</p><pre>git clone <a href="https://github.com/nishantapatil3/spire-federation-kind">https://github.com/nishantapatil3/spire-federation-kind</a></pre><p>Lets build our server and client</p><pre>./1-build.sh</pre><p>Create two clusters using kind, follow instructions as below<br><a href="https://kind.sigs.k8s.io/docs/user/quick-start/#creating-a-cluster">https://kind.sigs.k8s.io/docs/user/quick-start/#creating-a-cluster</a></p><p>Lets name them as kind-1 and kind-2 in our case</p><pre># Create kind clusters<br>kind create cluster --name kind-1<br>kind create cluster --name kind-2</pre><pre># Export kubeconfig to desired location<br>mkdir -p ~/kubeconfigs<br>kind get kubeconfig --name=kind-1 &gt; ~/kubeconfigs/kind-1.kubeconfig<br>kind get kubeconfig --name=kind-2 &gt; ~/kubeconfigs/kind-2.kubeconfig</pre><p>Once you have two clusters running, deploy metallb and spire components</p><p>We need to install metallb load balancer to allow external traffic to reach pods inside our kubernetes cluster by external ip services</p><pre># Apply the helm chart</pre><pre>helm template helm/metallb-system --set globalPrefix=&quot;255&quot; | kubectl apply --kubeconfig $cluster1 -f -<br>helm template helm/metallb-system --set globalPrefix=&quot;254&quot; | kubectl apply --kubeconfig $cluster2 -f -</pre><p>After we have load balancers in our clusters we have to deploy spire-server and spire-agent</p><p>To do that lets apply helm charts using below cmd</p><pre>helm template helm/spire --set trustDomain=cluster1.com --set federatesWith[0].trustDomain=cluster2.com --set federatesWith[0].address=172.17.254.1 --set federatesWith[0].port=8443 | kubectl apply --kubeconfig $cluster1 -f -</pre><pre>helm template helm/spire --set trustDomain=cluster2.com --set federatesWith[0].trustDomain=cluster1.com --set federatesWith[0].address=172.17.255.1 --set federatesWith[0].port=8443 | kubectl apply --kubeconfig $cluster2 -f -</pre><p>Next, bootstrap spire certificates with each other such that the workloads (server and client) can get the certificate from either trust domains to establish connection</p><pre>$ ./2-bootstrap.sh<br>Setting clusters kubeconfig /home/ubuntu/spire-federation-kind/lab_clusters.sh<br>Bootstrap certificate from cluster1 to cluster2<br>bundle set.<br>Bootstrap certificate from cluster2 to cluster1<br>bundle set.</pre><p>Run the following command to create spire entries so that spire server can know which trust domain certificate the workload can fetch from to connect to its target</p><p>Client gets two certificates and iterates over them to find the right certificate</p><pre>$ ./3-register.sh<br>Setting clusters kubeconfig /home/ubuntu/spire-federation-kind/lab_clusters.sh<br>-------------------------<br>Registering workload: server<br>Entry ID      : 12c07f5f-8629-4654-9961-dd6dbeba577e<br>SPIFFE ID     : spiffe://cluster1.com/server<br>Parent ID     : spiffe://cluster1.com/spire-agent<br>Revision      : 0<br>TTL           : default<br>Selector      : k8s:sa:server-service-account<br>FederatesWith : spiffe://cluster2.com</pre><pre>-------------------------<br>-------------------------<br>Registering workload: client<br>Entry ID      : 8082ce59-4bca-43a5-83d5-006890f822b5<br>SPIFFE ID     : spiffe://cluster2.com/client<br>Parent ID     : spiffe://cluster2.com/spire-agent<br>Revision      : 0<br>TTL           : default<br>Selector      : k8s:sa:client-service-account<br>FederatesWith : spiffe://cluster1.com</pre><pre>-------------------------</pre><p>Deploy server in cluster1 and client in cluster2</p><pre>$ kubectl apply -f helm/server.yaml --kubeconfig $cluster1<br>serviceaccount/server-service-account created<br>service/stock-quotes-service created<br>deployment.apps/stock-quotes-service created<br>$ kubectl apply -f helm/client.yaml --kubeconfig $cluster2<br>serviceaccount/client-service-account created<br>service/broker-webapp created<br>deployment.apps/broker-webapp created</pre><p>Now we can Port forward the client pod to your localhost:8080 using kubectl, get the client-pod name and replace below cmd</p><pre>Example:<br>kubectl port-forward broker-webapp-85574f4585-cxvxg 8080:8080 — kubeconfig $cluster2</pre><pre>$ kubectl port-forward broker-webapp-85574f4585-q92x4 8080:8080 --kubeconfig $cluster2<br>Forwarding from 127.0.0.1:8080 -&gt; 8080<br>Forwarding from [::1]:8080 -&gt; 8080<br>Handling connection for 8080<br>Handling connection for 8080<br>Handling connection for 8080<br>Handling connection for 8080<br>Handling connection for 8080</pre><p>Open up a browser to <a href="http://localhost:8080/quotes">http://localhost:8080/quotes</a> and you should see a grid of randomly generated phony stock quotes that are updated every 1 second.</p><figure><img alt="" src="images/1-zkp0tzv4-m-n35ughkzepw.png" /></figure><p>Server (left k9s) and client (right k9s), you can see that client fetched two certificated from cluster1.com and cluster.com, iterated over these certificates and picked the right one for the target</p><figure><img alt="" src="images/1-8gvhnuw_eguqjdwlq4i8hg.png" /></figure><p>Congrats you have successfully created spire federated clusters!!</p><h4>Conclusion</h4><p>What Spire can do?</p><ul><li>Secure microservices communication</li><li>Validates cryptographic service identities</li><li>Eliminates the need for secret management</li><li>Automatically issues, distributes, and renews short-live credentials</li><li>Reduces operational overhead associated with credential management</li></ul>
